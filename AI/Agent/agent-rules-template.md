# 核心工作流：渐进式上下文加载协议 (Progressive Context Loading Protocol)
在处理任何用户请求（无论是写代码、改 Bug 还是设计方案）时，**必须**严格执行以下两阶段上下文加载流程：

## 阶段一：强制全局基座加载 (Mandatory Global Base)
无论用户的需求多小，**第一步必须**读取并深刻理解以下全局指导文件。这是项目的"宪法"，不可违背：
1. `[项目索引文件路径]`：了解项目全貌、目录结构与协作边界。
2. `[全局规范文件路径]`：掌握 API 响应契约、全局异常处理、编码规范及代码分层标准。
3. `[数据字典/核心配置路径]`：掌握核心数据结构、字段约束、模型映射关系。

## 阶段二：按需渐进式模块加载 (On-Demand Progressive Loading)
在吸收全局规范后，分析当前任务的具体业务领域。根据索引文件中的说明，**仅精准读取**与当前任务直接相关的子模块文件，**严禁**一次性加载整个文档目录以防上下文溢出和注意力稀释。
- **示例 A**：若任务为"[业务场景 A]"，则**仅加载** `[对应模块文件路径]`。
- **示例 B**：若任务为"[技术场景 B]"，则**仅加载** `[对应参考文件路径]`。
- **示例 C**：若任务为简单实体/模型生成，则**无需加载**子模块，仅依赖阶段一的全局规范即可。

# 文件输出管理规范 (File Output Management)

Agent 在执行任务过程中产生的所有文件（包括中间临时文件、草稿文件和最终结果文件），除非用户提示词中**显式指定了其他保存路径**，否则**必须全部保存在项目根目录 `[项目根路径占位符]` 下**。

## 核心约束
1. **路径优先级**：用户显式指定路径 > 项目根目录下的自适应子目录 > **禁止**散落于系统临时目录或工作区之外。
2. **目录自组织**：Agent 应根据任务类型和文件用途，自适应地在项目根目录下创建具有语义化命名的子文件夹进行分类整理。目录命名应清晰反映文件用途（如 `output/reports/`、`temp/drafts/`、`artifacts/build/` 等），确保文件结构清晰可追溯。
3. **临时文件生命周期**：对于明确标注为"临时"的中间文件，Agent 应在任务完成后主动清理；若需保留，应在文件命名中体现临时属性（如 `temp_` 前缀或 `.tmp` 后缀），以便用户后续识别和清理。

# 执行动作与思考外化 (Think Aloud)
在生成最终代码或回答之前，你**必须**在回复的开头使用 `<context_loading>` 标签，简要说明你的加载策略，让用户知道你没有"盲目生成"。

格式如下：
```

1. 已加载全局基座：[列出已读取的全局文件]。
2. 任务分析：当前任务涉及 [具体业务/技术点]。
3. 渐进加载：按需读取 [具体子模块/参考文件路径]。若无需额外加载则填"无"。
4. 核心约束提取：[列出 1-3 条从文档中提取的、与当前任务最相关的强制规范]。
</context_loading>
```

# 绝对禁止事项 (Red Lines)
1. **禁止脱离规范造轮子**：如果全局规范中已定义 [如：统一返回格式/异常处理]，**绝对禁止**自行定义其他格式或绕过现有机制。
2. **禁止核心模型裸奔**：**绝对禁止**将 [如：JPA Entity / 数据库模型] 直接作为 API 响应返回，必须在服务层转换为 DTO，以防止数据泄露和循环引用。
3. **禁止低效查询**：在编写涉及关联关系的查询时，**必须**主动使用 [如：JOIN FETCH / 批量加载]，禁止在循环中调用查询。
4. **禁止破坏数据结构**：生成的模型字段、类型、注解必须与数据字典严丝合缝。若发现文档设计有缺陷，必须先向用户提出修改建议，而不是擅自修改代码。
5. **禁止上下文贪吃**：严禁盲目读取整个项目代码，必须依赖规范文档作为主要上下文来源。
